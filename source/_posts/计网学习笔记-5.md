# 网络层：控制平面
如前所述，转发表（或流表）既可以采用**分布式控制**，也可以采用**逻辑集中式控制**。

在**分布式控制**模式下，每台路由器都运行路由选择协议，同时具备转发和路由选择功能。每台路由器中的路由选择组件会与其他路由器的对应组件交换信息，协同计算并更新本地转发表。

而在**逻辑集中式控制**（如 SDN 架构）中，转发表由一个逻辑上集中的控制器统一计算并下发。此时，路由器中仅保留一个**控制代理**，该代理只能与控制器通信，并严格按照其指令执行转发操作。这些代理之间**不能直接交互**，也**不参与转发表的计算**，仅作为控制器指令的执行端。

## 路由选择算法
路由选择本质上明显是一个图论问题。根据是否能在算法运行前获取全网的拓扑信息和链路开销，可分为**集中式路由选择算法**和**分布式路由选择算法**：  
- **集中式算法**假设每个节点都掌握整个网络的完整拓扑和所有链路的开销信息，一般被称作链路状态算法（Link-State, LS）；
- **分布式算法**则仅要求每个节点了解与其直接相连的邻居及对应链路的开销，通过迭代交换路由信息逐步收敛，一般被称作距离向量算法（Distance-Vector, DV）。

另一种分类方式是根据算法是否动态，静态算法使用固定路径，不随网络状态变化；而动态算法能够根据网络流量负载或拓扑变化实时调整路由，适应性更强。

此外，还可按算法是否负载敏感进行分类。在负载敏感算法中，链路开销会随当前拥塞程度动态调整，从而引导流量避开瓶颈；然而，当前因特网广泛使用的路由协议（如 OSPF、BGP）均为**负载迟钝**的，即链路开销通常为静态配置，不随实时流量变化。

### 链路状态路由选择算法
明显是用 Dijkstra，我们重点探讨比较复杂的**路由振荡**现象。在某些网络拓扑中，路由选择策略可能导致多台路由器基于相同的链路状态信息**同步地反复切换路径**。例如，所有路由器可能在某一时刻一致选择顺时针方向转发流量，下一刻又同时切换至逆时针方向，导致流量在两条路径间来回“跳跃”。这种集体行为不仅引发路径抖动，还会造成链路负载剧烈波动，甚至加剧拥塞。

一种直观的解决思路是将链路开销设为静态值，使其不随流量负载变化，从而切断负载与路由之间的反馈循环。然而，这种“负载迟钝”的方式虽然能抑制振荡，却丧失了对网络拥塞的动态感知能力，显然不可取。

另一种思路是避免所有路由器同时执行 LS 算法，但由于它们在接收到相同事件（如链路状态更新）后容易自发同步计算，反而更容易形成集体振荡。为此，实践中普遍采用在发送链路状态通告时引入随机延迟的方法，使各路由器在不同时间点触发更新与路径重计算。这种轻微的异步化有效打破了同步性，使得路由可以更加快速、稳定地收敛。

### 距离向量路由选择算法
DV 算法是一种**迭代的、异步的、分布式**算法：

- **分布式**体现在每个节点仅与一个或多个直接邻居交换信息，基于本地计算更新自己的距离向量，并将结果再分发给邻居；
- **迭代**体现在该过程持续进行，直到所有节点的距离向量收敛，即邻居之间不再有新的信息需要交换；
- **异步**则指各节点无需在统一时刻进行计算，而是根据收到邻居更新的时机独立、自主地触发本地计算，不要求时钟同步。

一个重要的关于最短路径的性质是 Bellman-Ford 方程：

$$d_x(y)\ =\ \min_v{c(x,v)+d_v(y)}$$

基于此，每个节点维护其到其余各个节点的估计代价，然后在收到邻居的信息后进行更新并通知所有邻居即可，本质上还是 Floyd 算法。

然而，DV 算法虽然概念简单，却容易引发一个经典问题——**计数到无穷**。当某条链路失效或开销突增时，路由器可能误判其邻居仍持有到达目的网络的更短路径，从而基于错误信息更新自身的距离向量；而邻居又会依据这一更新反向修正自己的路由表，双方陷入相互“传递错误”的循环，导致到目的网络的估计代价缓慢递增，迟迟无法收敛，甚至一路增长至协议定义的“无穷大”（例如 RIP 中的 16）。

为缓解这一问题，实际部署中引入了多种抑制机制，其中最典型的是**毒性逆转**。其核心思想是：若某路由器通过邻居 A 学习到一条通往目的网络的路由，则在向 A 发送路由更新时，会将该目的网络的距离显式标记为无穷大（即“毒化”该路由），因为 A 再经由自己到达该目的网络显然不是最优路径。这一机制阻断了错误路由信息相邻节点间的往复传播，但对于涉及多节点的环路仍无能为力，更多节点就需要触发更新、抑制计时器等额外机制来辅助抑制。

对比来看，LS 算法的优势在于其收敛速度快与全局一致性强。每台路由器获取全网拓扑信息，独立运行最短路径算法计算到所有目的地的最优路径。一旦链路状态变化被通告，全网能在较短时间内同步更新，几乎不会出现计数到无穷等问题。其缺点是报文复杂度较高：需要周期性向全网广播链路状态，且每台路由器必须存储完整拓扑并独立计算，计算与存储开销较大，因此更适合中大型网络。

DV 算法的主要优势则是实现简单、报文负担小。路由器仅需周期性地向邻居交换自己的各个估计距离，无需了解全局网络架构，也不依赖复杂的全局计算，因此适用于规模较小、结构稳定的网络。然而，DV 的收敛速度较慢，在链路开销变化时可能出现路由振荡与计数到无穷等问题。为改善其健壮性，实际协议通常引入毒性逆转等机制以防止错误传播。

因此，实际中两种算法往往结合使用。

## OSPF
我们之前将网络简化为一组互联路由器的集合，但在实践中，这种简化往往不成立。一方面，网络规模庞大，难以让所有路由器运行同一套全局路由算法；另一方面，每个 ISP 通常希望自主控制其内部路由策略，并对外隐藏网络的内部拓扑和策略细节。理想情况下，一个组织应能**独立地运行和管理自己的网络**，同时又能**无缝地与外部网络互连互通**。正因如此，催生了自治系统（Autonomous System, AS）的概念和域间路由机制。

一个自治系统由全局唯一的 AS（ASN）标识，同一自治系统内的所有路由器运行相同的内部路由选择算法，并彼此交换完整的拓扑或路由信息。这种在自治系统内部使用的路由选择协议被称为**自治系统内部路由选择协议**。

AS 内部最常用的路由选择协议是**开放最短路径优先**（Open Shortest Path First, OSPF）。其中“开放”意味着该协议规范是公开可用的，“最短路径优先”则表明其基于链路状态并采用 Dijkstra 算法计算最短路径。OSPF 是一种**链路状态路由协议**：每台路由器通过泛洪机制向 AS 内所有其他路由器广播其直连链路的状态信息，从而使得每台路由器都能构建出整个自治系统的完整拓扑图，并在此基础上运行 Dijkstra 算法，计算到所有目的地的最低开销路径。

各条链路的开销由网络管理员手动配置。若所有链路开销均设为 1，则 OSPF 实现的是**最少跳数路由**；开销也可根据链路带宽设置（通常与容量成反比），以避免流量过多地使用低带宽链路。需要注意的是，OSPF 本身并不强制规定开销的具体设定策略，而是提供一种机制：在给定链路权重集合的前提下，自动计算出总开销最低的路径。这种灵活性使 OSPF 能够适应多种网络设计目标。

使用 OSPF 时，路由器会向 AS 内**所有**其他路由器广播链路状态信息。每当一条链路的状态发生变化（如链路失效或恢复），路由器会立即泛洪更新后的链路状态通告（LSA）；即使网络拓扑未发生变化，也会周期性地发送 LSA 以确保数据库同步。OSPF 报文直接由 IP 承载（协议号为 89），因此 OSPF 必须自行实现可靠传输和链路状态泛洪机制，包括确认、重传和顺序控制等，以保证链路状态数据库的一致性。此外，OSPF 还通过定期发送 Hello 报文来检测邻居是否可达，从而判断链路是否正常运行。同时，OSPF 路由器通过与相邻路由器交换数据库描述报文，对比并同步链路状态数据库，仅传输必要的差异信息，从而快速、精确地对齐数据库内容，确保彼此拥有一致的网络拓扑视图。

## BGP
如前所述，我们还需要一个 AS 间的路由选择协议。事实上，因特网中所有的 AS 运行相同的 AS 间路由选择协议，称为**边界网关协议**（Broder Gateway Protocol, BGP）。

### BGP 的作用
在 BGP 中，分组并非被精确地送到某个具体的目的主机地址，而是根据一个 **CIDR 化的前缀**来决定路径，每个前缀代表一个子网或一组子网的聚合。因此，BGP 路由表中的条目以**前缀**（如 `138.16.68.0/22`）作为键，对应的值通常是该路由器用于转发匹配流量的下一跳信息，也就是下一个路由器的 IP 地址。BGP 的工作可以概括为两个主要步骤：
1. **学习路由信息**：
   每台路由器从其他 AS 的邻居那里获得各种网络前缀的可达信息，并把自己负责的网络（比如自己的子网）向互联网其他地方广播。
2. **选择最佳路径**：
   当一个前缀有多条路径时，路由器会在本地运行 **BGP 路由选择算法**，按照优先级挑出最合适的一条，作为到达该前缀的“最佳路由”。

### 通告 BGP 路由信息
在 BGP 中，我们将路由器分为**网关路由器**和**内部路由器**。BGP 通过在路由器之间建立 TCP 连接（端口 179）来交换路由可达性信息。连接分为两类：**外部 BGP**（eBGP），用于不同 AS 之间的网关路由器；**内部 BGP**（iBGP），用于同一 AS 内部的路由器之间。eBGP 会话通常建立在直连的 AS 边界路由器之间，而 iBGP 会话则不一定对应物理链路，常通过逻辑全互联或路由反射器等方式在 AS 内分发路由信息。

当某个 AS 宣告一个前缀的可达性时，该信息首先通过 eBGP 传入相邻 AS，再由接收 AS 的边界路由器通过 iBGP 将完整 AS 路径（如 `AS₂ AS₃ x`）通告给本 AS 内所有其他路由器。这样，每个路由器不仅获知目的前缀的存在，还知道到达该前缀所经的 AS 序列。若存在多条路径（如经不同邻居 AS 到达同一前缀），BGP 将根据策略和路径属性（如 AS 路径长度）选择最优路由。一个简单的例子如图所示：

### 确定最好的路由
如前所述，从一台路由器到某个目的子缀可能存在多条可用路径，而因特网中的 BGP 路由器通常会收到来自不同邻居的多条可达性通告。为此，路由器会根据 BGP 的**路由属性**在这些路径中进行选择，并据此配置其转发表。其中最重要的两个属性是 **AS-PATH** 和 **NEXT-HOP**。

一条 BGP 路由由**目的前缀**、**AS-PATH** 和 **NEXT-HOP** 共同定义。**AS-PATH** 记录了该路由从源 AS 到当前通告者所经过的自治系统序列，事实上，每当一个 AS 转发该路由时，都会将自己的 ASN 插入 AS-PATH 前端。路由器利用此属性不仅用于路径选择（通常优先选择 AS-PATH 较短的路径），还可以用于**环路检测**，只需要在 AS-PATH 包含本 AS 的编号时丢弃该路由即可。**NEXT-HOP** 则指明下一跳路由器的 IP 地址，也即 AS-PATH 起始 AS 中网关路由器面向本 AS 的接口地址。尽管该地址不属于本地 AS，但它所在的子网必须直连本 AS，以便数据包能正确转发。通过结合这两个属性，路由器能够在多条 BGP 路由中确定路由，并将选中的路径映射到具体的输出端口。

#### 热土豆路由选择
热土豆路由选择是最简单的 BGP 路由决策策略，其核心思想是在多条可达外部前缀的路径中，选择通往对应 NEXT-HOP 所需 AS 内部开销最小的那条。换句话说，路由器会优先考虑“尽快把流量送出本自治系统”，而不关心 AS 外部到最终目的地的路径开销——就像扔掉一个烫手的热土豆，越快脱手越好。~~好形象啊~~

具体而言，当一台路由器通过 BGP 学习到通往同一外部前缀 $x$ 的多条路由时，它会结合 AS 内部路由协议（如 OSPF）的信息，计算到达各个 NEXT-HOP 的内部最短路径开销，并在其转发表中为前缀 $x$ 选择对应的最优输出接口。值得注意的是，即使同一 AS 内的两台路由器收到相同的 BGP 路由，也可能因到不同网关的内部开销不同而选择不同的出口路径。这体现了热土豆路由的“本地最优”特性，它优先最小化本 AS 内的资源消耗，但不一定能实现端到端的全局最优。

#### 路由器选择算法
在实践中，BGP 采用一种比热土豆路由更精细的多级路由选择算法。当路由器针对某一目的前缀学习到多条有效路由时，会按顺序应用一系列消除规则，直到选出唯一最优路径：

1. **本地偏好**：优先选择本地偏好值最高的路由。该属性由 AS 管理员配置，用于实施内部策略，完全在本 AS 内生效；  
2. **AS-PATH 长度**：在本地偏好相同的情况下，选择 AS-PATH 最短（即穿越 AS 数最少）的路由，这有助于减少路径跳数并隐式优化端到端性能；  
3. **热土豆路由**：若前两项仍无法区分，则采用热土豆策略，选择到达 NEXT-HOP 所需 AS 内部开销最小的路径；  
4. **BGP 路由器标识符**：实在不行就选择来自具有最小 BGP ID 的邻居的路由。

在前述例子中，路由器 1b 虽然根据热土豆策略可能更倾向于经 $AS_2$ 转发（因为其 NEXT-HOP 更近），但由于不经过 $AS_2$ 的路径具有更短的 AS-PATH，BGP 会在路径选择过程中优先选择后者。这种设计使 BGP 能在保证策略一致性的同时，倾向于选择路径更短、传播更高效的路由。

### IP 任播
除了作为 AS 间的路由选择协议外，BGP 还常被用于实现 IP 任播。任播的核心思想是将同一个 IP 地址分配给地理位置分散的多台服务器，那么当用户发起请求时，BGP 的路由选择机制会将流量导向路径最优的节点，这样就可以有效地降低访问延迟。

在具体实现上，服务提供商会让分布在不同地理位置的多台服务器通过 BGP 同时通告相同的 IP 前缀。当路由器收到多条指向该前缀的 BGP 路由时，会根据路由选择算法选出最优路径，从而将流量引导至距离最近的服务器。虽然这一机制在理论上也适用于 CDN，但实际中 CDN 通常避免采用 IP 任播，因为 BGP 路由的动态变化可能导致同一 TCP 连接的数据包被分散转发到不同服务器，进而破坏连接状态。相比之下，DNS 系统广泛应用任播机制。当用户向某个逻辑地址发起查询时，BGP 会自动将请求路由到地理位置最近的根服务器实例，从而显著提升解析速度与系统的可靠性。
