# 网络层：数据平面
在网络中，每台主机和路由器都包含网络层的功能模块。也正因如此，网络层协议是整个协议栈中最为复杂且最富挑战性的部分。通常来说，网络层可以被划分为**数据平面**和**控制平面**：数据平面负责决定数据报如何被转发，而控制平面则负责确定路由策略。在第四章中，我们主要学习**数据平面**的相关内容。

## 网络层概述
### 转发和路由选择
网络层的作用表面上极为简单，就是将分组从一台发送主机传输到另一台接收主机。然而，要实现这一目标，网络层需要依赖两种核心功能：

- **转发**：当一个分组到达路由器的某条输入链路时，路由器需将其移至合适的输出链路。转发是**数据平面**所实现的唯一功能。在某些情况下，分组也可能被路由器丢弃，例如来自已知恶意主机的流量，或目的地址属于被禁止访问的主机。
- **路由选择**：当分组从发送方流向接收方时，网络层需确定其所采用的路由（或路径），用于计算这些路径的算法称为**路由选择算法**。路由选择功能由**控制平面**实现。

路由器中一个至关重要的组件是**转发表**。当一个分组到达路由器时，路由器会检查该分组首部中的一个或多个字段，并利用这些字段的值作为关键字在转发表中进行查找。转发表中的每一项通常包含匹配条件和对应的输出链路，路由器根据匹配结果确定该分组应从哪条输出链路转发出去。

转发表的配置方式主要有两类：**传统分布式方法**和**软件定义网络**（Soft-ware-Defined Networking, SDN）。在传统网络中，每台路由器既负责数据平面的转发，也运行控制平面的路由协议，通过与其他路由器交换网络状态信息，自主计算路由并本地生成转发表，控制与数据功能紧密耦合于同一设备。而在 SDN 架构中，控制平面被集中化，由远程的 SDN 控制器统一计算路径并主动向各交换机或路由器下发转发表项，数据平面仅负责按表转发，从而实现**控制平面与数据平面的逻辑分离**。理论上，转发表甚至可以完全由人工静态配置（例如通过命令行设置静态路由），无需动态协议参与，这进一步说明转发行为本身并不依赖本地控制逻辑，因此控制与转发在功能上是可分离的。

### 网络服务模型
理论上，网络层可以提供多种多样的服务，例如：确保分组可靠交付、在限定时延内完成交付、保证分组有序到达、提供最小带宽保障，以及支持安全传输等。然而，正如我们在传输层中所讨论的，因特网的网络层仅提供一种服务：“尽力而为”服务，也就是说，因特网既不能保证端到端时延，也不保证有最小的带宽。这一术语听起来几乎像是“没有服务”的委婉说法。相比之下，其他网络体系结构（如ATM）确实实现了更强大的服务模型，提供明确的服务质量保证。尽管如此，因特网基于尽力而为服务辅以充足的带宽供给，已被实践证明“足够好”，足以支撑从网页浏览、视频流媒体到实时通信等极其广泛的应用场景。

  <img src="/illustrations/计网笔记4/1.png" alt="路由器体系结构" width="90%">
  <figcaption>路由器体系结构</figcaption>
</figure>

- **输入端口**：输入端口是路由器中连接物理链路的硬件接口，承担了多项关键功能。其主要职责包括：  
  1. **物理层终结**：接收并解析来自入链路的物理信号，完成比特流到数字数据的转换（对应最左侧功能模块）；  
  2. **链路层处理**：与入链路远端设备协同完成数据链路层协议操作，如帧同步、差错检测、流量控制等（对应中间功能模块）；  
  3. **转发查找**：提取分组首部中的关键字段（如目的 IP 地址），在转发表中执行高速查找，确定对应的输出端口，并将分组送入交换结构。对于需要本地处理的控制分组（如路由协议报文），则将其转发至路由选择处理器（对应最右侧功能模块）。  
  在实际路由器中，输入端口的类型和数量差异很大，可支持从低速串行链路到高速光纤接口（如 100 Gbps 甚至更高）的广泛速率范围，以适应不同网络场景的需求。
- **交换结构**：交换结构负责将输入端口连接到它的输出端口。
- **输出端口**：输出端口负责接收来自交换结构的分组，将其暂存于输出队列中，并执行必要的链路层（如帧封装、差错编码）和物理层（如信号调制、时钟同步）功能，最终将分组通过输出链路发送出去。当通信链路为**双向**时，输入端口与输出端口通常成对集成在同一块**线路卡**上，共享物理接口资源，以提高硬件集成度和转发效率。
- **路由选择处理器**：在传统路由器中，该处理器负责运行路由选择协议，维护路由表及相关的链路状态或可达性信息，并据此生成转发表供数据平面使用。在 SDN 架构的路由器中，其不再本地计算路由，而是与远程 SDN 控制器通信，接收由控制器集中计算并下发的转发表项，并将这些表项安装到输入端口的转发硬件中。此外，路由选择处理器通常还承担网络管理功能。

路由器的输入端口、输出端口和交换结构几乎总是采用专用硬件实现，这是因为它们需要以极高的速度处理海量分组，转发决策通常必须在纳秒级完成，而通用软件处理难以满足如此严苛的性能要求。相比之下，路由选择处理器通常运行在通用计算机硬件和操作系统之上，其处理时延在毫秒量级即可接受，这使得路由器能够灵活地执行各种动态路由协议、响应网络拓扑变化，并支持丰富的网络管理功能。

可以看到，输入端口的线路端接功能与链路层处理模块共同实现了各输入链路对应的物理层和数据链路层功能；随后，通过查询转发表确定相应的输出端口，使到达的分组经由交换结构被正确转发至目标输出端口。该转发表由路由选择处理器本地计算并更新，或在 SDN 架构下由远程控制器下发。为支持高速转发，转发表会通过独立总线从路由选择处理器复制到各线路卡（在路由器体系结构图中以虚线表示）。借助这些本地副本，每个输入端口可独立完成转发决策，无需调用路由选择处理器，从而显著提升转发效率。

最简单的情形是要为一个 32 位的 IP 地址进行转发。若采用暴力实现，转发表需包含多达 40 亿个条目，显然不可行。实际上，我们采用基于目的地址**前缀**的匹配方式，并遵循**最长前缀匹配**原则。

假设转发表已经存在，理论上只需遍历所有表项即可找到匹配项，但由于路由器的转发决策必须在**纳秒级**时间内完成，这种线性查找方式远远无法满足高性能转发的需求。因此，实际系统采用专门的**快速查找算法**（如基于 trie 的结构、哈希等）来高效实现最长前缀匹配。与此同时，为尽可能缩短内存访问延迟，系统通常采用分层存储策略：完整的转发表因其规模庞大，存储在大容量、低成本的 **DRAM** 中，由路由选择处理器负责维护；而频繁访问或近期使用的表项则被缓存在高速的 **SRAM** 中，供输入端口在纳秒级时间内完成高速查找，从而兼顾容量与转发性能。

一旦通过查找确定了某分组的输出端口，该分组便可送入交换结构。但如果此时交换结构正被其他分组占用，则该分组可能会被暂时阻塞，必须在输入端口处排队，等待后续被及时调度并通过交换结构。

尽管“查找”在输入端口处理中通常被视为最关键的操作，但实际上接收端口还需完成多项其他必要任务：
1. 必须执行物理层和链路层的处理（如前所述）；
2. 必须检查分组的版本号、首部校验和以及 TTL 等字段（相关内容将在下一节讨论），并对校验和与 TTL 字段进行更新；
3. 还需更新用于网络管理的统计计数器（例如接收到的 IP 数据报总数）。

### 交换
交换结构负责连接输入端口与输出端口，但其实现方式多种多样。常见的交换技术如图所示：

<figure style="text-align: center;">
  <img src="/illustrations/计网笔记4/3.png" alt="常见交换技术" width="90%">
  <figcaption>常见交换技术</figcaption>
</figure>

- **经内存交换**：  
  分组由输入端口直接送入路由器的主内存，由路由选择处理器（或 CPU）读取分组首部、查找转发表，并将分组复制到对应输出端口的缓存中。这种方式结构简单、成本低，但受限于内存带宽和处理器速度，吞吐量较低，难以满足高速网络需求。
- **经总线交换**：  
  输入端口通过共享总线将分组直接发送至目标输出端口，无需经过主处理器或主内存。每个分组在总线上以带标签的形式广播，所有输出端口都能接收到该分组，但只有标签与自身地址匹配的输出端口才会接收并处理数据。这种方式虽避免了处理器瓶颈，但受限于总线带宽，同一时刻仅能传输一个分组，多个分组同时竞争总线时易产生冲突和阻塞，从而成为系统吞吐量的瓶颈。
- **经互联网络交换**：  
  为克服总线带宽的限制，现代路由器采用由多级交叉开关（如图中的纵横式交换机）构成的专用互联网络结构。这种结构能够在多个输入端口同时向不同输出端口发送分组时，实现无冲突的并发传输，即具备**非阻塞**特性。该方案显著提升了交换结构的并发能力和吞吐量，因此被广泛应用于高速核心路由器。然而，其硬件实现复杂，成本也相对较高。

## 网际协议
我们将关注点转向当今因特网网络层的关键特性及其核心协议：**网际协议**（IP）。目前，因特网上实际运行着两个 IP 版本。我们首先研究广泛部署的 **IP 版本 4**（IPv4），随后将深入探讨 IPv6，该版本已被设计用于逐步取代 IPv4。在此过程中，我们将重点学习**因特网编址**机制。

### IPv4 数据报格式
如前所述，网络层分组被称作数据报，IPv4 的数据报格式如图所示：

<figure style="text-align: center;">
  <img src="/illustrations/计网笔记4/5.png" alt="IPv4 数据报格式" width="50%">
  <figcaption>IPv4 数据报格式</figcaption>
</figure>

- **版本**：4 位字段，标识 IP 协议版本，IPv4 中该值为 4。  
- **首部长度**：4 位字段，指示 IPv4 首部长度（以 32 位字为单位），最小值为 5（即 20 字节），若含选项则更长。  
- **服务类型**：8 位字段，用于区分不同类型的数据报。
- **数据报长度**：16 位字段，表示整个 IP 数据报（首部 + 数据）的总字节数，理论上最大为 65,535 字节，一般不超过 1500 字节。  
- **标识**：16 位字段，用于唯一标识一个 IP 数据报，便于分片后的重组。  
- **标志**：3 位字段，控制分片行为，其中 MF 位表示“还有更多分片”，DF 位表示禁止分片。  
- **片偏移**：13 位字段，指示当前分片在原始数据报中的位置（以 8 字节为单位）。  
- **生存时间**：8 位字段，限制分组在网络中的最大跳数，每经过一跳路由器减 1，减到 0 时丢弃分组，防止无限循环。  
- **协议**：8 位字段，指明 IP 数据报载荷所使用的上层协议（如 6 表示 TCP，17 表示 UDP）。  
- **首部校验和**：16 位字段，用于检测 IPv4 首部在传输过程中是否出错（不包含数据部分）。  
- **源 IP 地址**：32 位字段，标识发送方的 IPv4 地址。  
- **目的 IP 地址**：32 位字段，标识接收方的 IPv4 地址。  
- **选项**：可变长字段（如有），用于网络测试、调试或特殊路由等（如记录路由、时间戳），因性能和安全原因现已很少使用，IPv6 中已经去掉。  
- **数据**：一般是运输层报文段，也可承载 ICMP 报文等。

### IPv4 数据报分片
由于并非所有链路层协议都能承载相同长度的网络层分组，每种链路都定义了一个**最大传送单元**（MTU），即该链路所能承载的最大 IP 分组长度。当 IP 数据报的长度超过某条链路的 MTU 时，就必须对其进行**分片**，并将每个分片分别封装在独立的链路层帧中进行传输。

这些片在到达目的地运输层前需要重新组装，但若在中间路由器上进行重组，会显著影响转发性能。依据**端到端原则**，IP 的设计者将重组责任放在通信的两端，而非网络中的中间节点。

为此，IPv4 首部引入了三个关键字段：**标识**、**标志**和**片偏移**。当源主机生成一个 IP 数据报时，会为其分配一个唯一的标识号；该标识号在分片过程中保持不变，目的主机据此判断哪些分片属于同一个原始数据报。由于 IP 本身是不可靠的，某些分片可能在传输过程中丢失。为标识最后一个分片，IPv4 使用标志字段中的 **MF**（More Fragments）：除最后一片外，所有分片的 MF 位均设为 1，最后一片设为 0。此外，**片偏移**字段指明了该分片在原始数据报中的起始位置（以 8 字节为单位），使目的主机能够正确排序并检测是否缺失分片，从而完成准确重组。

### IPv4 编址
虽然我们常说“主机或路由器发送和接收 IP 数据报”，但实际上所有的 IP 通信都是通过其**网络接口**完成的，通常来说，主机只有一个网络接口，而路由器则拥有多个接口，分别连接不同的网络。因此，IP 协议要求每个参与通信的**网络接口**必须配置一个 IP 地址。从技术角度看，IP 地址是与**接口**关联的，而不是与主机或路由器这个设备整体绑定。这意味着，一台多接口设备（如路由器）会拥有多个 IP 地址，每个接口对应一个，用于在其所连接的特定网络中进行通信。

每个 IPv4 地址长度为 32 比特（即 4 字节），因此理论上共有 $2^{32}$ 个可能的 IP 地址。这些地址通常采用**点分十进制记法**表示：将地址的 4 个字节分别转换为十进制数，并用句点分隔。例如，IP 地址 `193.32.216.9` 中，193 对应前 8 位，32 对应第二个 8 位，216 对应第三个 8 位，9 对应最后一个 8 位。其完整的二进制形式为：

```
11000001 00100000 11011000 00001001
```

然而，IP 地址并非可以随意自由选择。事实上，一个接口的 IP 地址中，有一部分必须由其所连接的子网决定，以确保网络层能够正确进行路由和编址。

在 IP 网络中，**子网**是指一组通过同一物理或逻辑链路互连的设备接口，这些接口共享相同的网络前缀。例如，一个典型的局域网可能包含若干主机和一个路由器的接口，它们都连接到同一个交换机或无线接入点，构成一个广播域，因此属于同一个子网。

为该子网分配的地址通常采用**无类别域间路由**（Classless Inter-Domain Routing, CIDR），如 `223.1.1.0/24`。其中，`/24` 表示地址的前 24 位（即前三个字节）为**网络前缀**，用于唯一标识该子网；剩余 8 位为**主机部分**，用于标识子网内的各个接口。因此，该子网中所有有效接口地址的形式均为 `223.1.1.x`（其中 $ x \in [1, 254] $），例如：
- 主机可使用 `223.1.1.1`、`223.1.1.2`、`223.1.1.100` 等；
- 路由器在该子网中的接口通常使用 `223.1.1.1` 或 `223.1.1.254` 等约定地址。

任何新加入该子网的设备都必须配置一个符合 `223.1.1.0/24` 前缀的 IP 地址，才能与该子网内的其他接口直接通信。事实上，每个子网都拥有独立的地址空间和专属的接口集合。

可以看到，当域外路由器转发目的地址位于该子网内部的数据报时，只需匹配前 24 位网络前缀，而无需关心具体的主机地址。这极大地减少了这些路由器转发表中的条目数量，从而显著提升了转发效率。

在 CIDR 被采用之前，IP 地址的网络部分长度被严格限制为 8、16 或 24 比特，这种编址方式称为**分类编址**。具体而言，网络前缀长度为 8 比特的称为 **A 类网络**，16 比特的为 **B 类网络**，24 比特的为 **C 类网络**。相应地，一个 IP 地址的网络部分必须恰好占用 1、2 或 3 个字节。

这种僵化的划分在实践中很快暴露出严重问题：  
- A 类网络可容纳约 1600 万个主机，但全球仅有 128 个，极易造成地址浪费；  
- C 类网络仅支持 254 个主机，对于中等规模组织又明显不足；  
- B 类网络（约 65,000 主机）数量有限，且无法灵活适配实际需求。

随着互联网的快速发展，越来越多的小型和中型组织需要接入网络，而传统的分类编址由于无法进行精细的地址划分，导致地址空间利用率低、路由表规模急剧膨胀。CIDR 有效地解决了这一问题，它允许网络前缀长度采用任意比特数，从而能够根据实际需求灵活分配地址块，显著提高地址利用率，并有效抑制路由表的过度增长。

此外，还有一个特殊的 IP 地址：当主机发送目的地址为 `255.255.255.255` 的数据报时，报文会被投递给**同一子网内的所有主机**，这被称为**受限广播**。通常情况下，路由器**不会**转发此类广播，以防止广播流量在网络中泛滥；但在某些特定配置下，路由器也可以选择性地将其转发到邻近子网，不过这种情况较为少见。

最后，我们来看看如何获取 IP 地址。主机和路由器接口的 IP 地址通常通过以下两种主要方式获取：

- **从 ISP 分配**：组织或家庭网络的公网 IP 地址一般由 ISP 分配。ISP 根据用户需求从其拥有的地址块中划出一个或多个地址（或子网），供用户边界路由器使用。
- **通过 DHCP 自动获取**：在局域网内部，主机通常使用**动态主机配置协议**（Dynamic Host Configuration Protocol, DHCP）自动获取 IP 地址。当设备接入网络时，会通过受限广播向 DHCP 服务器发送请求；服务器随后分配一个临时的 IP 地址（通常来自预设的私有地址池，如 `192.168.x.x`），在租约有效期内可尝试分配之前使用过的地址，并同时提供子网掩码、默认网关、DNS 服务器等网络配置参数，从而实现即插即用的网络接入。

### 网络地址转换
**网络地址转换**（NAT, Network Address Translation）是一种在边界路由器（如家庭或企业网关）上实现的技术，用于将内网使用的私有 IP 地址转换为公网 IP 地址，从而允许多台内网设备共享一个（或少量）公网 IP 地址访问因特网。

RFC 1918 定义了三段**私有地址空间**，这些地址不能在公网路由，仅用于内部网络，可自由使用而无需向 ISP 申请：
- `10.0.0.0/8`（10.0.0.0 – 10.255.255.255）  
- `172.16.0.0/12`（172.16.0.0 – 172.31.255.255）  
- `192.168.0.0/16`（192.168.0.0 – 192.168.255.255）

NAT 的核心原理是利用运输层的端口信息“虚拟”出多个独立的通信通道。它结合 DHCP 机制，通过维护一张 **NAT 转换表**，记录内网主机的（私有 IP 地址，端口号）与（公网 IP 地址，端口号）之间的映射关系。例如，当内网主机 `192.168.1.10:5000` 向外发起 TCP 连接时，NAT 会将其源地址替换为公网地址（如 `203.0.113.1:6000`），并将该映射存入转换表；当外部响应返回时，NAT 根据数据报的目标端口查找表项，将目的地址和端口还原为对应的内网主机地址和端口。通过这种方式，NAT 能在仅有一个（或少量）公网 IP 地址的情况下，支持大量内网主机并发访问互联网。

由于 NAT 隐藏了内网拓扑，且通常拒绝未经请求的入站连接，外部主机无法主动与内网主机建立通信。这对 P2P、VoIP、视频会议等依赖双向连接的应用构成了显著挑战。针对此类问题，常见的技术解决方案包括 **NAT 穿越**和通用即插即用协议等。

尽管 NAT 并非 IP 协议最初的设计内容，但它在缓解 IPv4 地址枯竭、提供基本的网络隔离方面发挥了重要作用，并成为 IPv4 向 IPv6 过渡阶段的重要过渡机制。诚然，NAT 将部分网络层的职责转移到了传输层，这在设计上引发了一些争议；然而不可否认的是，它在实际应用中极为高效、实用。

与 IPv4 数据报格式相比，IPv6 作出了多项重要改进：最显著的是 IP 地址长度从 32 比特扩展到 **128 比特**，极大地增加了地址空间；同时引入了**任播地址**，允许一个地址被多个接口共享，网络将数据报路由到“最近”的那个实例。此外，IPv6 采用**固定长度的 40 字节首部**，去除了 IPv4 中的分片字段、选项字段和首部校验和，使首部结构更简洁，便于硬件高速处理。值得一提的是，IPv6 新增了**流标签**字段，用于标识属于同一数据流的分组，为实时应用（如音视频传输）提供更好的服务质量支持。这些优化共同提升了路由效率、可扩展性和对新兴应用的适应能力。

可以注意到，IPv6 移除了一些 IPv4 中的字段，主要出于简化首部结构、提升转发效率和适应现代网络环境的考虑：

- **首部校验和**  
  现代链路层和传输层协议已具备完善的差错检测机制，IP 层的校验功能因此变得冗余。IPv6 取消首部校验和后，路由器在转发时无需因 TTL 更新而重新计算校验和，显著降低了处理开销，提升了转发速度。数据完整性仍由上下层协议保障，不影响端到端可靠性。
- **选项字段**
  IPv4 的选项字段长度可变，导致首部长度不固定，增加了路由器的解析和处理复杂度。IPv6 则将这些可选功能移入**扩展首部**，仅在需要时附加在基本首部之后。各个扩展首部通过**下一个首部**字段彼此连接，形成一条可灵活延展的首部链。大多数路由器在转发时只需处理基本首部即可，后续扩展首部通常由目标主机解析，从而既保留了协议的可扩展性，又使基本首部固定为 40 字节，结构简洁，便于硬件高速处理。
- **分片相关字段**
  在 IPv4 中，中间路由器可以对超过链路 MTU 的分组进行分片，但这增加了转发负担，且任一分片丢失都会导致整个数据报重传，效率低下。IPv6 将分片操作**限定在源主机**完成，发送方通过**路径 MTU 发现（PMTUD）**机制预先确定端到端路径的最小 MTU，并在源端进行分片（如有必要）。当路由器遇到过大的分组时，不再执行分片，而是返回一个“分组过大”的报文通知源主机调整分组大小。这样一来，路由器无需参与分片，转发逻辑更简单，性能更高，同时也减少了因分片丢失导致的重传风险。

最后，我们自然会好奇怎么从 IPv4 向 IPv6 迁移。显然，若规定从某一天起所有人都必须切换到 IPv6，这在全球范围内几乎不可能实现。互联网规模庞大、设备众多、协议栈复杂，IPv4 与 IPv6 的兼容性问题也无法一夜解决。因此，人们设计了多种过渡机制以实现平滑迁移。其中，**双栈**是最基本的方式：网络设备同时运行 IPv4 与 IPv6 协议栈，根据目标地址类型自动选择相应协议进行通信。与此同时，**隧道技术**被广泛用于在 IPv4 网络上传输 IPv6 流量，即将 IPv6 数据报封装进 IPv4 分组，在仅支持 IPv4 的中间网络中传递，抵达目标网络后再解封装为 IPv6。通过这些机制，IPv4 与 IPv6 能够长期共存并保持互通，使互联网得以在不中断既有服务的情况下逐步演进至全面的 IPv6 环境。

## 通用转发和 SDN
传统路由器的转发行为是**协议固定的**，只能根据目标 IP 地址查询转发表进行转发。而**通用转发**打破了这一限制，允许转发决策基于分组首部中的**任意字段组合**（如源/目的 IP 地址、端口号、协议类型、VLAN 标签，甚至应用层标识），并支持执行**多种动作**（如转发、丢弃、复制、修改字段、QoS 标记等）。

其核心机制是**“匹配-动作”**模型：  
- **匹配**：从分组首部提取一个或多个字段，与预设规则进行匹配；  
- **动作**：若匹配成功，则执行相应操作，如转发、丢弃或修改字段。

这一模型不再局限于“查 IP 转发表 → 转发”的单一逻辑，而是将数据平面转变为可编程的分组处理流水线，为实现灵活、细粒度的网络控制奠定了基础。

在此之上，SDN 进一步将控制平面与数据平面解耦，并将控制逻辑集中到一个逻辑上中心化的 **SDN 控制器**中。整个 SDN 架构通常分为三层：

- **数据平面**：由支持通用转发的交换机或路由器构成，仅负责按规则高速转发分组，不再运行传统的分布式路由协议；  
- **控制平面**：由一个或多个 SDN 控制器组成，掌握全网拓扑与实时状态，统一计算并维护转发规则，通过南向接口将规则下发至数据平面设备；  
- **应用平面**：运行在控制器之上的网络应用（如负载均衡、防火墙、流量工程等），通过北向 API 调用控制能力，实现高级网络服务与策略自动化。

这种架构实现了网络的集中管控与灵活编程，是现代可编程网络的核心基础。

# 奇怪的感想
计算机网络的发展好像是不断“抽象—固化—再抽象”的轮回过程。最早，网络功能完全由软件实现，灵活但性能差；然后为了追求速度与确定性，出现了专用硬件，一切都写死在芯片里；但随着需求变化加快，又出现了可编程交换芯片等方案，让人重新获得灵活性，各种解隅导致了超级灵活的各种语法。

似乎每当性能瓶颈被硬件解决，人们又开始追求灵活性；而当灵活性带来复杂与低效，又会回到简化和专用。
